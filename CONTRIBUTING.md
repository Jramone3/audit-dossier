# Contributing to Audit Dossier

Thank you for your interest in contributing to the Audit Dossier project! This repository documents security research, audit methodologies, and autonomous system resilience practices.

## How to Contribute

### 1. Reporting Issues
If you find inaccuracies, security concerns, or have suggestions for improvements:
- Open an Issue with clear title and description
- Include relevant context (links, code snippets, error messages)
- Use labels appropriately (bug, enhancement, documentation, security)

### 2. Submitting Documentation
We welcome contributions for:
- Case studies and security analysis
- Technical documentation and methodologies
- Research papers and findings
- Translation improvements

**Process:**
1. Fork the repository
2. Create a feature branch: `git checkout -b feature/your-contribution`
3. Make your changes following the style guide below
4. Commit with clear messages: `docs: add case study on CVE-XXXX`
5. Push to your fork and create a Pull Request

### 3. Style Guide

#### Markdown Files
- Use clear, professional language
- Include headers for section organization
- Add code blocks with appropriate language syntax highlighting
- Include references and links where applicable
- For technical content, provide examples or diagrams

#### File Organization
- Case studies go in `/docs/case-studies/`
- Technical documentation in `/docs/technical/`
- Assets (images, diagrams) in `/assets/`
- Translations in `/docs/{language}/`

#### Commit Messages
```
<type>(<scope>): <subject>

<body>

<footer>
```

Types: `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`

Example:
```
docs(security): add CVE-2025-55182 case study

Detailed analysis of the vulnerability and remediation strategies.
```

### 4. Content Guidelines

- **Accuracy**: All technical claims must be accurate and verifiable
- **Completeness**: Provide sufficient context for readers to understand concepts
- **Professionalism**: Maintain professional tone throughout
- **Security**: Never include sensitive credentials, API keys, or private information
- **Attribution**: Always credit sources and cite original research

### 5. Pull Request Process

1. Ensure your PR addresses one topic (avoid mixing unrelated changes)
2. Update documentation if you're changing features
3. Include a clear description of what and why you changed it
4. Link any related issues
5. Wait for review and respond to feedback
6. Once approved, your PR will be merged

### 6. Code of Conduct

- Be respectful and professional
- Provide constructive feedback
- Welcome diverse perspectives
- Report inappropriate behavior to: remi.bunker.sys@proton.me
